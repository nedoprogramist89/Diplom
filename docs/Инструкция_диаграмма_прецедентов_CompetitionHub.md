# Инструкция: диаграмма прецедентов WEB-приложение CompetitionHub (формат как в примере)

**Основа:** **«Пояснительная записка»** — подраздел с **диаграммой прецедентов** (Рисунок 1), описание ролей и сценариев во **внешней спецификации**; **«Методические материалы»** — требование к **диаграмме вариантов использования (UML)**; фактическая реализация — репозиторий **CompetitionHub** (приложения **`accounts`**, **`competitions`**, **`hackathons`**, проект **`config`**, панель **`/admin/`**, REST **`/api/`**).

**Формат примера:** ориентир — структура **инструкции к схеме тестирования** (`docs/Инструкция_схема_тестирования_из_документа_3.md`): сверху ясный **заголовок предмета**, далее **смысловые блоки** с **маркированными пояснениями**; отличие для **диаграммы прецедентов**: внутри границы системы — **овалы use case**, снаружи — **акторы** (палочные человечки), а не сотни прямоугольников тестов. Чтобы **объём по смыслу** был сопоставим с примером, ниже для каждой роли перечислено **что входит в обобщённый прецедент** (эти строки при переносе на рисунок **сворачиваются** в один овал и **примечание** под рисунком).

**Цель:** получить **читаемую** диаграмму (**ориентир 12–22 овала** на одном листе вместе с общими «вход / выход / регистрация»); детализацию (URL, отдельные кнопки CRUD) вынести в **примечание**, **таблицу приложения к ПЗ** или **второй уровень** (пакет / вложенная диаграмма — по согласованию с руководителем).

**Рекомендация по листу:** **А4 альбом** или **А3 альбом** при добавлении пакетов «Соревнования / Хакатоны / API»; при отдельном рисунке для **Клиент API** — **второй лист** или **правый фрагмент** того же листа. Масштаб вставки в Word: **55–70 %** ширины страницы либо шрифт подписей овалов **9–11 pt** (проверить распечаткой).

---

## 1. Корень рисунка и граница системы

В центре листа (или чуть левее центра, если справа оставляете зону API):

- **Прямоугольник границы системы** с подписью **CompetitionHub** (или «WEB-приложение CompetitionHub»).
- **Подпись под рисунком по ГОСТ:** **«Рисунок … – Диаграмма прецедентов WEB-приложения CompetitionHub»** (или «… вариантов использования …» — как требует кафедра).

**Акторы** разместить **по периметру** слева/справа/снизу:

- **Гость**, **Участник**, **Организатор**, **Жюри**, **Персонал** (панель `is_staff`), опционально снизу — **Клиент API**.

**Обобщение актора** (по желанию, «как в учебнике UML»): от **Организатора** и **Жюри** стрелка обобщения к **Участнику** с пояснением «наследует сценарии участника», чтобы **не дублировать** все овалы участника на организаторе и жюри. Если преподаватель требует **полный** набор овалов у каждого актёра — **продублировать** связи участника у организатора и жюри **без** стрелки наследования.

---

## 2. Общие прецеденты (несколько акторов)

Центральные **овалы**, к которым ведут линии от **Гостя**, **Участника**, **Организатора**, **Жюри**, **Персонала** (вход в систему и выход):

- **Войти в систему** — маршрут `accounts/login/`.
- **Выйти из системы** — `accounts/logout/`.
- **Зарегистрироваться в системе** — `accounts/register/` (в основном **Гость**; после успеха — роль **Участник**).

**Компактный вариант (как на схеме с малым числом овалов):** пункты **«Войти»** и **«Зарегистрироваться»** заменить **одним** овалом **«Вход или регистрация в системе»**; тогда в разд. 3 у **Гостя** не дублировать отдельные овалы входа и регистрации.

**Примеры связей UML:**

- **Войти в систему** `<<include>>` **Проверить учётные данные**.
- **Зарегистрироваться в системе** `<<include>>` **Создать учётную запись с ролью «Участник»**.
- **Войти в систему** (или объединённый овал входа/регистрации) `<<extend>>` **Показать сообщение об ошибке входа** (неверный пароль).

## 3. Детализация по ролям (что «сворачивается» в один овал)

Ниже каждый маркер — **не отдельный овал на рисунке**, а **строка для примечания / таблицы ПЗ**, если сверху выбран **обобщённый** прецедент из второго столбца.

### 3.1. Гость — прецедент **«Просмотр публичного контента»** покрывает:

- Просмотр главной страницы («центр событий»), ссылок на итоги и архив.
- Просмотр списка соревнований с поиском и фильтрами (статус, аудитория, уровень, возраст, свободные места — по реализации).
- Просмотр карточки соревнования и списка задач без входа (если разрешено правилами страницы).
- Просмотр карточки задачи (условие, варианты без раскрытия правильного ответа — по типу задания).
- Просмотр страниц **итогов** и **архива** (`/results/`, `/archive/`).
- Просмотр списка и карточки **хакатонов** (публичная часть).
- Просмотр **публичного профиля** пользователя по ссылке (`/accounts/profile/<username>/`).
- Просмотр **сертификата** участника (если выделять отдельным маленьким овалом — только по требованию руководителя; иначе — в примечании к этому прецеденту).

### 3.2. Гость — прецедент **«Восстановить доступ к аккаунту»** покрывает:

- Запрос сброса пароля по e-mail.
- Переход по ссылке из письма, ввод нового пароля, подтверждение (цепочка `password_reset` / `confirm` / `complete` в `accounts`).

### 3.3. Гость — прецедент **«Вход или регистрация в системе»** (или два овала «Войти» / «Зарегистрироваться») покрывает:

- Открытие формы регистрации, заполнение логина, e-mail, пароля, подтверждения, отправка формы.
- Открытие формы входа, ввод логина и пароля, успешный вход и редирект в кабинет.

### 3.4. Участник — прецедент **«Управление профилем и уведомлениями»** покрывает:

- Просмотр и редактирование данных профиля (в т. ч. организация).
- Смена **темы оформления** (светлая / тёмная).
- Смена пароля из кабинета; **тестовое письмо** SMTP (если выведено в интерфейсе).
- Список **уведомлений**, отметка «прочитано», переход по уведомлению.
- Маршруты вида `/accounts/profile/`, `…/edit/`, `…/theme/`, `…/password/…`, `…/notifications/…`.

Отдельно на диаграмме часто оставляют общий овал **«Выйти из системы»** (разд. 2).

### 3.5. Участник — прецедент **«Участие в соревновании»** покрывает:

- Регистрация на соревнование на этапе **«Регистрация»** и **отмена** регистрации при допустимых правилах.
- Просмотр блока **«Мои активные участия»** на главной после входа.

### 3.6. Участник — прецедент **«Работа с решениями и результатами»** покрывает:

- Открытие формы **отправки решения** в этапе **«Идёт»** с учётом расписания задачи (текст, выбор варианта, соответствие, файл — по типу).
- Просмотр **«Мои решения»** по задаче.
- Просмотр **таблицы результатов** соревнования.

**Связи `<<include>>`** при отправке решения (к вложенному или текстовому условию): **Проверить этап «Идёт»**; **Проверить регистрацию на соревновании**.

### 3.7. Участник — прецедент **«Участие в хакатоне и командах»** покрывает:

- Регистрация и отмена на **хакатоне** в допустимой фазе.
- Хаб **команд**: создание команды, заявка, одобрение / отказ, выход, роспуск, передача роли капитана (по реализации `hackathons`).
- **Чат** (общий и канал организатор — команда), если предусмотрено интерфейсом.

**Роль капитана** отдельным актором **не выделять**, если методичка не требует.

### 3.8. Участник — связь с гостем

- Одна ассоциация к **тому же** овалу **«Просмотр публичного контента»**, что и у **Гостя** (без второго такого же овала).

### 3.9. Организатор — прецеденты **поверх** возможностей участника (обобщённо)

Ниже — **развёрнутый перечень** в стиле «много строк — один овал» (как листья дерева в инструкции к схеме тестирования, но для **примечания** к диаграмме, а не для отдельных овалов).

#### 3.9.1. Овал **«Управление своими соревнованиями»** — в примечании перечислить:

- Переход к форме **создания** турнира с главной или со списка соревнований.
- Заполнение названия, описания, статуса, дат, аудитории, уровня, квоты, возрастных ограничений (по полям формы).
- Сохранение нового турнира и появление карточки в списке.
- **Редактирование** своего турнира и сохранение изменений.
- **Удаление** своего турнира с подтверждением в диалоге.
- **Быстрая смена этапа** на карточке (`quick_status`), если выведена в интерфейсе.

#### 3.9.2. Овал **«Управление задачами соревнования»** — в примечании перечислить:

- Список задач турнира; **добавление** задачи с выбором предмета и типа задания.
- Текст условия, баллы, порядок; **варианты ответа** и отметка верных (для типов с выбором).
- Пары **«соответствие»**; загрузка **файла материала** организатором (если тип/форма поддерживают).
- Настройка **расписания** открытия и закрытия приёма ответов (если реализовано).
- **Редактирование** и **удаление** задачи с подтверждением.

#### 3.9.3. Овал **«Управление объявлениями»** — в примечании перечислить:

- Создание объявления на карточке турнира.
- Редактирование текста и признака закрепления (если есть).
- Удаление объявления с подтверждением.

#### 3.9.4. Овал **«Модерация участников и оценка решений»** — в примечании перечислить:

- Просмотр **списка участников** своего турнира.
- Переход к **всем решениям** по задаче; открытие формы **оценки** (баллы, статус, комментарий).
- Сохранение оценки и отображение обновлённых данных в списке.
- При наличии в проекте — просмотр **истории изменения оценки** (`Grade events`).

**Связь `<<include>>`:** для оценки — **Проверить права создателя соревнования** (или персонала).

#### 3.9.5. Овал **«Экспорт отчётности по соревнованию (CSV)»** — в примечании перечислить:

- Выгрузка **результатов** (места, участники, баллы).
- Выгрузка **участников** (если кнопка предусмотрена).
- Выгрузка **решений** (если кнопка предусмотрена).

#### 3.9.6. Овал **«Управление своими хакатонами»** — в примечании перечислить:

- Создание, редактирование и удаление **своего** хакатона.
- Смена **этапа** хакатона; просмотр списка участников хакатона.
- Назначение участника в команду (`assign_team` и аналоги по реализации).
- Участие в **канале организатор — команда** в чате (если выделено в ТЗ как отдельная функция — всё равно остаётся внутри этого овала, без нового овала «Чат» на главной диаграмме).

### 3.10. Жюри — дополнительно к участнику

**«Просмотр и оценка работ участников»** — в примечании перечислить:

- Просмотр списка участников **любого** турнира (в пределах прав роли `jury`).
- Открытие карточки задачи при допустимых ограничениях видимости.
- Переход к **всем решениям** по задаче; форма **оценки** (баллы, статус, комментарий); повторное открытие и корректировка (если разрешено).

**«Просмотр результатов и экспорт CSV»** — в примечании перечислить:

- Таблица **результатов** турнира.
- Те же виды **выгрузки CSV**, что доступны организатору в вашей сборке (результаты / участники / решения — по факту кнопок в интерфейсе).

**Не соединять** актора **Жюри** с овалами управления турнирами, задачами, объявлениями и хакатонами **как организатор**.

### 3.11. Персонал (`/admin/`) — крупные овалы вместо десятков CRUD по моделям

#### Овал **«Администрирование пользователей и прав»** — в примечании:

- Список пользователей; создание и правка записи пользователя.
- Назначение **роли**, **активности**, принадлежность к **группам**.

#### Овал **«Администрирование данных соревнований»** — в примечании:

- Справочники **предметов** и **типов заданий**.
- Модели приложения `competitions`: соревнования, задачи, варианты, соответствия, участия, решения, события оценок, объявления — **все операции**, которые зарегистрированы в `admin.py` и доступны staff.

#### Овал **«Администрирование хакатонов»** — в примечании:

- Модели приложения `hackathons`: хакатоны, команды, участники, регистрации, сообщения чата — по факту регистрации в админке.

#### Овал **«Управление уведомлениями»** — in-app уведомления (CRUD по правам).

#### Овал **«Просмотр журнала аудита»** — только чтение `LogEntry` (если модель зарегистрирована).

Дашборд и фильтры списков — **в примечании**, не отдельными овалами.

### 3.12. Клиент API — два овала

**«Аутентификация и профиль по API»** — в примечании перечислить:

- Получение пары JWT: `POST /api/auth/token/` (или эквивалент в вашей версии).
- Обновление access-токена: `POST /api/auth/token/refresh/` (если включено).
- Регистрация через API: `register/` при наличии маршрута.
- Текущий пользователь: `me/`.

**«Доступ к соревнованиям по API»** — в примечании перечислить:

- Чтение и изменение **соревнований** и **задач** согласно `competitions/api_urls.py`.
- **Участия**, **регистрация** на турнир, **решения**, **результаты** — методы и коды ответов в таблице приложения к ПЗ.

**`<<include>>`:** для защищённых вызовов — **«Передать Bearer-токен»** (или вложенный use case).

## 5. Связи `<<include>>` и `<<extend>>` (рекомендуемый минимум)

| От | Тип | К | Смысл |
|----|-----|---|--------|
| **Работа с решениями и результатами** (отправка) | include | Проверить этап «Идёт» | бизнес-правило |
| То же | include | Проверить регистрацию на соревновании | целостность |
| **Модерация…** / **Просмотр и оценка…** (жюри) | include | Проверить права (создатель / жюри / персонал) | безопасность |
| **Управление своими соревнованиями** | include | Проверить роль «Организатор» или персонал | |
| **Доступ к соревнованиям по API** | include | Аутентификация по JWT | защищённые методы |
| **Войти** или **Вход или регистрация** | extend | Показать сообщение об ошибке | опционально |
| **Зарегистрироваться** или **Вход или регистрация** | extend | Отправить письмо | если настроено |

## 6. Как нарисовать в редакторе (чтобы схема была крупной и читаемой)

1. Включить **сетку** и направляющие; ориентация листа — **альбом**.
2. Разместить **акторов** слева и справа (и **Клиент API** снизу, если нужен).
3. Нарисовать **границу системы** с подписью **CompetitionHub**; внутри — **только обобщённые овалы** из разд. 2–5 (ориентир **не более 18–22** овалов на листе при объединённом входе/регистрации).
4. Опционально: **пунктирные пакеты** «Соревнования», «Хакатоны», «Учётная запись» вокруг групп овалов **без** вложенных мелких дубликатов.
5. Провести **ассоциации**; стереотипы **`<<include>>`** / **`<<extend>>`** — выборочно (разд. 5), не «звёздное небо».
6. В **draw.io**: библиотека **UML** — фигуры use case и actor; единый шрифт подписей **10–12 pt**; выравнивание овалов по колонкам.
7. В **StarUML** / **Visual Paradigm**: модель **Use Case Diagram**, элементы Actor / Use Case / System Boundary.
8. Под рисунком — **примечание**: «Крупные прецеденты расшифрованы в таблице … приложения … / в разделе … ПЗ».
9. При добавлении в проект новых `path(...)` — либо **дополнить примечание** к существующему овалу, либо (новая зона смысла) добавить **один** новый обобщённый прецедент.
