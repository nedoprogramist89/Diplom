# Инструкция: схема взаимодействия компонентов (CompetitionHub)

Документ для подготовки **рисунка уровня «архитектура / компоненты»** (в пояснительной записке — аналог «Рисунок 24 – Схема взаимодействия компонентов приложения»): не диаграмма классов UML и не DFD, а **кто с кем связан** при обработке запроса пользователя.

---

## 1. Цель схемы

Показать цепочку: **клиент (браузер или API-клиент)** → **точка входа URL** → **представления (логика страниц и API)** → **модели и проверка прав** → **СУБД**, а также участие **шаблонов** и **статики** в ответе пользователю по веб-интерфейсу.

Схема отвечает на вопросы: *куда попадает HTTP-запрос*, *кто вызывает кого*, *где хранятся данные*.

**Эталон оформления** — схема в стиле «Схема взаимодействия компонентов CompetitionHub»: сверху **Клиент**, снизу **База данных**, внутри одной крупной рамки **«Приложение Django (CompetitionHub)»** — блоки и стрелки, как в вашем примере к рисунку 24 (см. также подпись к рисунку в пояснительной записке).

## 2. Эталонная структура (как на примере)

Ниже — **точный перечень блоков и логики связей**, который нужно воспроизвести при построении рисунка (имена подписей на схеме — как в примере).

### 2.1. Внешние уровни

1. **Клиент** — над рамкой приложения (браузер и/или REST-клиент).
2. Общая рамка **«Приложение Django (CompetitionHub)»** — все внутренние компоненты находятся внутри неё.
3. **База данных** — под рамкой приложения (СУБД).

Рекомендуется сохранить **PNG-эталон** вашего рисунка в репозитории (например `docs/images/Схема_взаимодействия_эталон.png`), чтобы при доработках ПЗ сверять подписи и стрелки с утверждённым вариантом.

### 2.2. Внутри рамки «Приложение Django» (сверху вниз)

| № | Блок на схеме | Смысл |
|---|----------------|--------|
| 1 | **Маршрутизатор** | Единая точка входа запросов внутри приложения (соответствует корневому `urlpatterns` в `config/urls.py`). |
| 2 | **URL учётных записей** | Маршруты приложения `accounts`. |
| 3 | **URL соревнований** | Маршруты веб-интерфейса соревнований (`competitions`, корень сайта). |
| 4 | **API маршруты** | Префикс REST API (`config/api_urls.py` и вложенные `include`). |
| 5 | **Панель управления** | Административный интерфейс (`/admin/`). |
| 6 | **Представления учётных записей** | `accounts/views.py` (и при необходимости связанные модули). |
| 7 | **Представления соревнований** | `competitions/views.py`. |
| 8 | **API представления** | `accounts/api_views.py`, `competitions/api_views.py`. |
| 9 | **Шаблоны** | Каталог `templates/` (рендер HTML для веб-части). |
| 10 | **Проверка прав** | `competitions/permissions.py`, `hackathons/permissions.py`, проверки в представлениях. |
| 11 | **Модели пользователей** | `accounts/models.py` (в т.ч. `User`). |
| 12 | **Модели соревнований** | `competitions/models.py` (и связанные сущности турниров/задач). |
| 13 | **Сериализаторы** | `accounts/serializers.py`, `competitions/serializers.py` (только цепочка API). |

### 2.3. Стрелки по эталону (обязательная логика)

- **Клиент ↔ Маршрутизатор** — двусторонняя связь (запрос от клиента и ответ клиенту).
- **Маршрутизатор** → **URL учётных записей**, **URL соревнований**, **API маршруты**, **Панель управления** — распределение по префиксам, как в `config/urls.py`.
- **URL учётных записей** → **Представления учётных записей**; **URL соревнований** → **Представления соревнований**; **API маршруты** → **API представления**.
- **Представления учётных записей**, **Представления соревнований**, **Панель управления** → **Шаблоны** (веб-ответ HTML).
- **Представления учётных записей**, **Представления соревнований**, **API представления**, **Панель управления** → **Проверка прав**.
- **Представления учётных записей**, **API представления**, **Панель управления** → **Модели пользователей**.
- **Представления соревнований**, **API представления**, **Панель управления** → **Модели соревнований**.
- **API представления** → **Сериализаторы** (только эта ветка).
- **Модели пользователей** и **Модели соревнований** → **База данных** (на эталонной схеме именно эти два блока опускают стрелки к БД; ORM не выносится отдельным прямоугольником).

### 2.4. Расширение эталона (если нужно полное соответствие коду)

В репозитории дополнительно есть раздел **«Хакатоны»** (`hackathons/`). Если его нужно показать на схеме курсового/дипломного объёма:

- добавьте блоки **URL хакатонов** и **Представления хакатонов** по аналогии с соревнованиями;
- от **Маршрутизатора** проведите стрелку к **URL хакатонов** (как в `path('hackathons/', include('hackathons.urls'))`);
- **Представления хакатонов** свяжите с **Шаблонами**, **Проверкой прав**, моделями хакатонов (отдельный блок **«Модели хакатонов»** — `hackathons/models.py`) и при необходимости — с **Базой данных** (если на рисунке явно показываете хранение; иначе можно объединить подписью «прочие модели → БД» по согласованию с руководителем).

## 4. Пошаговый порядок работы (по эталону)

1. Нарисовать **Клиента**, рамку **«Приложение Django (CompetitionHub)»**, **Базу данных**.
2. Внутри рамки сверху разместить **Маршрутизатор**; провести **двустороннюю** связь **Клиент ↔ Маршрутизатор**.
3. От маршрутизатора вниз — четыре блока: **URL учётных записей**, **URL соревнований**, **API маршруты**, **Панель управления**.
4. Под ними — три блока представлений: **Представления учётных записей**, **Представления соревнований**, **API представления** (стрелки от соответствующих URL).
5. Ниже разместить **Шаблоны**, **Проверка прав**, **Модели пользователей**, **Модели соревнований**, **Сериализаторы**; провести стрелки **строго как в п. 2.3** (в т.ч. сериализаторы только от API представлений).
6. От блоков **«Модели пользователей»** и **«Модели соревнований»** — стрелки только к **Базе данных**.
7. Сверить список с `config/urls.py`; при добавлении хакатонов — расширить схему по п. 2.4.

## 6. Где взять примеры кода для ООП (инкапсуляция, наследование, полиморфизм)

Ниже — **файлы собственного кода проекта** (без виртуального окружения), которые удобно цитировать в ПЗ или показывать преподавателю рядом с рисунками 21–23.

### 6.1. Инкапсуляция

Смысл: **сокрытие деталей** за интерфейсом модуля или объекта (данные + методы, проверки внутри класса).

| Где смотреть | Что показать |
|--------------|--------------|
| `competitions/permissions.py` | Функции `can_*` и миксины: внешнему коду достаточно вызвать проверку, внутренняя логика ролей сосредоточена в модуле. |
| `accounts/models.py` | Класс `User`: поля роли, темы; метод `get_role_description()` — работа с данными пользователя через метод модели. |
| `competitions/models.py` | У сущности `Task` (и связанных моделей) — методы/свойства вроде `is_multiple_choice()`, `is_matching()`: поведение «типа задачи» инкапсулировано в модели. |
| `competitions/forms.py`, класс `SolutionSubmitForm` | Методы `__init__` и `clean`: настройка полей и валидация внутри формы, а не в шаблоне. |

### 6.2. Наследование

Смысл: **класс-потомок** расширяет или специализирует **базовый** класс Django или ваш миксин.

| Где смотреть | Что показать |
|--------------|--------------|
| `accounts/models.py` | `class User(AbstractUser):` — пользователь как расширение стандартной модели Django. |
| `competitions/permissions.py` | `CanEditCompetitionMixin(UserPassesTestMixin)`, `CanGradeSolutionsMixin`, … — наследование от миксина проверки доступа Django. |
| `competitions/views.py` | Классы представлений с множественным наследованием, например `TaskCreateView(LoginRequiredMixin, CanEditCompetitionMixin, CreateView)` — цепочка миксинов + базовый `CreateView`. |
| `hackathons/views.py` | Аналогично: `HackathonCreateView(LoginRequiredMixin, CanCreateHackathonMixin, CreateView)`, `HackathonUpdateView(..., UpdateView)`. |
| `competitions/api_views.py` | Классы API на базе `generics.*APIView` из Django REST framework (например `ListCreateAPIView`, `RetrieveUpdateDestroyAPIView`) — наследование от фреймворка. |

### 6.3. Полиморфизм

Смысл: **один и тот же интерфейс** (один класс представления или одна форма) ведёт себя **по-разному** в зависимости от типа данных (здесь — тип задачи / полей формы).

| Где смотреть | Что показать |
|--------------|--------------|
| `competitions/forms.py`, `SolutionSubmitForm.__init__` | Ветвление по `task.is_multiple_choice()`, `task.is_matching()` и `task_slug`: разный набор полей и виджетов для разных типов заданий. |
| `competitions/views.py`, `SolutionSubmitView.form_valid` | Ветвление по типу задачи: множественный выбор, соответствие, сравнение с `expected_output` — разная логика автопроверки в одном методе. |
| `competitions/auto_grade.py` | Функции `compute_mc_multi_score`, `compute_mc_single_score`, `compute_match_score` и т.п. — разные алгоритмы оценки для разных видов ответа. |
| `competitions/api_views.py` | Приём решения через API: фрагменты с вызовом тех же `compute_*` при разных типах задачи (аналогично веб-представлению). |

## 7. Согласованность с текстом ПЗ

Фраза в духе: *«Клиент обращается к `urls.py`; представления взаимодействуют с моделями и БД; для проверки прав используется модуль `permissions`»* должна **буквально соответствовать** нарисованным блокам и стрелкам.

После правок кода перепроверьте, что на схеме нет устаревших имён файлов (например, если маршруты перенесли в другой `urls.py`).
