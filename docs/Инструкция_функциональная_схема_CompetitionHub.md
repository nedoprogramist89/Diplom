# Инструкция: функциональная схема программы (CompetitionHub)

Документ для подготовки **рисунка к подразделу 2.3.5** пояснительной записки (аналог **«Рисунок 32 – Функциональная схема»**): не диаграмма классов UML, не DFD и не перечень файлов `*.py` ([структурная схема](Инструкция_структурная_схема_CompetitionHub.md)), а **кто какие функции выполняет** в системе соревнований и как это связано с **хранением данных**.

---

## 1. Цель схемы

Показать **декомпозицию задачи** по **ролям (акторам)** и **блоку хранения данных**: какие группы действий доступны участнику, организатору, жюри, администратору и как все роли опираются на **базу данных** при выполнении функций.

Схема отвечает на вопросы: *какие функции у какой роли*, *чем полномочия ролей отличаются*, *где на схеме отражено хранение данных*.

**Эталон оформления** — как в абзаце раздела **2.3.5** пояснительной записки: блоки **Участник**, **Организатор**, **Жюри**, **Администратор**, **База данных**; внутри каждой роли (кроме БД) — перечень функций из **2.2.1** (роли и возможности). Рекомендуется сохранить **PNG-эталон** рисунка (например `docs/images/Функциональная_схема_эталон.png`) для сверки при правках ПЗ.

**Не путать с:** [схемой взаимодействия компонентов](Инструкция_схема_взаимодействия_компонентов_CompetitionHub.md) (рис. 24 — маршрутизатор, представления, файлы), [диаграммой прецедентов](Инструкция_диаграмма_прецедентов_CompetitionHub.md) (рис. 1 — овалы прецедентов), [схемой пользовательского интерфейса](Инструкция_схема_пользовательского_интерфейса_CompetitionHub.md) (рис. 33 — навигация по страницам).

## 2. Эталонная структура (как в ПЗ к рисунку 32)

Ниже — **перечень блоков и логики связей**, которые нужно воспроизвести на рисунке (подписи согласовать с **2.2.1** и **2.3.5**).

### 2.1. Внешний контур (уровень 1)

Один блок-заголовок над или вокруг зоны ролей, например: **«WEB-приложение CompetitionHub»** или **«Функции системы соревнований»**. Он задаёт **первый уровень** детализации по методичке.

### 2.2. Блоки уровня 2 (роли и хранилище)

| № | Блок на схеме | Смысл |
|---|----------------|--------|
| 1 | **Участник** | Пользователь с ролью `participant`: базовые сценарии турнира и профиль. |
| 2 | **Организатор** | Роль `organizer`: всё участника + управление **своими** соревнованиями, задачами, оценкой, CSV. |
| 3 | **Жюри** | Роль `jury`: всё участника + оценка и просмотр по **любым** соревнованиям, **без** создания/редактирования турниров и задач. |
| 4 | **Администратор** | Персонал (`is_staff`): панель `/admin/`, все сущности, журнал аудита. |
| 5 | **База данных** | СУБД (PostgreSQL / SQLite): хранение данных, с которыми работают функции ролей. |

### 2.3. Содержимое блоков уровня 3 (функции внутри роли)

Под каждым блоком **Участник**, **Организатор**, **Жюри**, **Администратор** разместите **список функций** (вложенные мелкие прямоугольники, маркированный текст в фигуре или отсылка «см. таблицу N»):

**Участник:** регистрация и выход; вход; главная и список соревнований (поиск, фильтр); карточка соревнования и список задач; регистрация на турнир и отмена (этап «Регистрация»); карточка задачи; отправка решения (этап «Идёт», тип задания задаёт форму ответа); свои решения и таблица результатов; профиль (данные, тема, пароль, восстановление по email, тест SMTP).

**Организатор:** всё перечисленное у участника; создание / редактирование / удаление **своих** соревнований; задачи, варианты ответа, объявления; участники **своих** турниров; все решения по задачам **своих** турниров, баллы, статус, комментарий; результаты и **экспорт CSV**.

**Жюри:** всё у участника; участники **любого** соревнования; все решения по любой задаче, оценка; результаты и CSV; **ограничение:** нет CRUD по соревнованиям и задачам (подпись или иконка «запрещено» в легенде).

**Администратор:** панель `/admin/`; пользователи (создание, роли, активность); соревнования, задачи, варианты, участия, решения, объявления, справочники, группы; **журнал аудита** (просмотр).

**База данных:** без внутреннего списка таблиц на этой схеме; подпись «хранение пользователей, турниров, задач, решений…».

### 2.4. Стрелки по эталону (обязательная логика)

- От блоков **Участник**, **Организатор**, **Жюри**, **Администратор** → **База данных** — использование хранилища при выполнении функций (допустима **одна** совокупная стрелка от группы «роли» к БД, если так читабельнее).
- Связь **Организатор** → **Участник** или **Жюри** → **Участник** — **не обязательна**; полномочия «всё участника» лучше пояснить **текстом в легенде** или дублированием списка функций, чтобы не перегрузить стрелками.
- При необходимости показать **поток сценария** между ролями (редко): например пунктир «после публикации турнира участник регистрируется» — только если руководитель требует показать взаимодействие ролей, а не только состав функций.

### 2.5. Расширение эталона (хакатоны, гость, API)

Если в объёме работы есть **хакатоны** и **публичный доступ без входа**:

- добавьте блок **Гость** (просмотр публичного контента, переход к регистрации/входу);
- добавьте блок **Хакатоны** или включите функции хакатона в списки **Участник** / **Организатор** с подзаголовком «хакатон»;
- при отдельном акценте на **REST API** допустим узел **«Клиент API (JWT)»** с функциями «получение данных / отправка решений по API» и стрелкой к **База данных** через подпись «через приложение» — по согласованию с руководителем, чтобы не дублировать [рис. 24](Инструкция_схема_взаимодействия_компонентов_CompetitionHub.md).

Сверка с [инструкцией по диаграмме прецедентов](Инструкция_диаграмма_прецедентов_CompetitionHub.md) — обязательна, если на схеме появляются новые актёры.

## 4. Пошаговый порядок работы (по эталону)

1. Зафиксировать **уровень 1** — заголовок системы (п. 2.1).
2. Разместить **пять блоков уровня 2** из таблицы п. 2.2 (или добавить Гость/API по п. 2.5).
3. Заполнить **уровень 3** — списки функций из п. 2.3 под каждой ролью (или вынести в таблицу под рисунком с отсылкой «см. таблицу N»).
4. Провести стрелки к **База данных** по правилам п. 2.4; оформить **легенду** (наследование полномочий участника, запрет жюри на CRUD турниров).
5. Сверить с **рисунком 1** (прецеденты) и с **2.3.5** — один смысл с текстом абзаца под рисунком 32.
6. Вставить рисунок в ПЗ, подпись **«Рисунок 32 – Функциональная схема»**; при необходимости добавить **таблицу** (п. 6).
7. Убедиться, что рисунок **не дублирует** структурную схему (файлы) и **не заменяет** [схему пользовательского интерфейса](Инструкция_схема_пользовательского_интерфейса_CompetitionHub.md) (рис. 33 — страницы).

## 6. Таблица к схеме и вариант «ООП» по методичке

Если требуют таблицу **№ — имя — физический смысл — свойства — действия**, используйте роли как **логические объекты**:

| № | Имя (на схеме) | Физический смысл | Свойства / ограничения | Основные действия (функции) |
|---|----------------|------------------|-------------------------|-----------------------------|
| 1 | Участник | Зарегистрированный пользователь-участник турниров | Роль `participant` | см. п. 2.3, блок Участник |
| 2 | Организатор | Создатель и ведущий своих соревнований | Роль `organizer`, только свои турниры | см. п. 2.3, блок Организатор |
| 3 | Жюри | Проверяющий без редактирования турниров | Роль `jury` | см. п. 2.3, блок Жюри |
| 4 | Администратор | Персонал панели управления | `is_staff` | см. п. 2.3, блок Администратор |
| 5 | База данных | СУБД | PostgreSQL / SQLite | Хранение и выдача данных |

**Макеты размещения блоков** (на выбор, как для наглядности): **дорожки** (колонка на роль), **звезда** (центр — название системы, лучи — роли), **дерево** (корень цель → роли → функции).

## 8. Проверка перед сдачей

- [ ] Есть **не менее трёх** уровней: система → роли (+ БД) → функции внутри роли (или уровень 3 в таблице с отсылкой из подписи к рисунку).
- [ ] Отражены **все роли** из **2.2.1**, для проекта с хакатонами — согласовано с **2.5** настоящей инструкции.
- [ ] Видно **отличие организатора и жюри** (свои турниры / любые соревнования без CRUD турниров).
- [ ] Блок **База данных** есть, стрелки от ролей к БД согласованы с п. 2.4.
- [ ] Нет смешения с **структурной** (имена `views.py`) и **схемой взаимодействия** (маршрутизатор).
