# Инструкция: логическая схема базы данных (CompetitionHub)

Документ для подготовки **рисунка к подразделу 2.3.2** пояснительной записки (**«Рисунок 29 – Логическая схема данных (сущность–связь)»**): **инфологическая** модель предметной области — **сущности**, **атрибуты** (без типов СУБД), **связи** и **кратности** («один-ко-многим», «многие-ко-многим» через ассоциативную сущность).

**Источник истины по проекту:** модели Django в `accounts/models.py`, `competitions/models.py`, `hackathons/models.py`. Физические таблицы и типы полей — в подразделе **2.3.3** пояснительной записки (рис. 30) и в приложении Г (`docs/postgres_schema.sql` — расширенный SQL-вариант, **не обязан** совпадать с ORM один в один).

---

## 1. Цель схемы и отличия от других рисунков

| Вид схемы | Вопрос, на который отвечает |
|-----------|-----------------------------|
| **Логическая (ER, инфологическая)** | *Какие объекты предметной области и как они связаны по смыслу?* Имена сущностей на **русском** (или двуязычно: «Пользователь / User»). |
| **Физическая** | *Как это хранится в PostgreSQL?* Имена **таблиц**, **типы**, **PK/FK**, **UNIQUE**, индексы. |
| **DFD** | *Какие потоки данных между процессами и хранилищами?* Не заменяет ER. |
| **Диаграмма классов (UML)** | *Какие классы, методы, наследование в коде?* Другой уровень абстракции. |
| **Рис. 25 «Данные в базе»** | Скрин или обзор **физических** таблиц; логическую схему **не дублирует**, а дополняет. |

**Не включать на логической схеме (если кафедра не требует явно):**

- служебные таблицы Django (`django_session`, `django_migrations`, `auth_permission` и т.д.) — вынести в сноску «инфраструктура фреймворка»;
- детали JWT/blacklist, если в дипломе API не акцентирован;
- каждое поле `AbstractUser` (достаточно блока атрибутов у сущности **Пользователь**).

**Включить обязательно** (по тексту ПЗ, разд. 2.2.2): пользователь, соревнование, задача, вариант ответа, участие, решение, объявление, справочники **Предмет** и **Тип задания**; для диплома с хакатонами — подсистему **Хакатон** (отдельная зона на рисунке).

## 2. Нотация на рисунке

Рекомендуется **нотация «воронья лапка» (Crow’s Foot)** или **Chen** (ромбы связей) — **одну** выбрать и не смешать на одном листе.

| Элемент | Как рисовать | Подпись на рисунке |
|---------|--------------|-------------------|
| Сущность | Прямоугольник | **Пользователь**, **Соревнование**, … |
| Атрибут | Овал или список внутри прямоугольника | `email`, `статус`, … — **5–12 ключевых** на сущность, остальное в таблице в ПЗ |
| Первичный ключ | Подчёркнутый атрибут или `id` | `id` |
| Связь | Линия с кратностью на концах | `1` — `N`, `1` — `1` |
| Ассоциация M:N | Отдельная сущность-связка | **Участие** (пользователь ↔ соревнование) |

**Подписи связей** — глаголом в смысле предметной области: *«создаёт»*, *«содержит»*, *«регистрируется на»*, *«сдаёт»*, *«оценивает»* (на линии или в ромбе).

**Цвет / зоны (рекомендация):**

- синяя рамка — `accounts`;
- зелёная — `competitions` (соревнования по задачам);
- оранжевая — `hackathons`;
- серые прямоугольники — справочники (**Предмет**, **Тип задания**).

## 3. Состав сущностей (перечень для рисунка 29)

### 3.1. Ядро: учётные записи и уведомления (`accounts`)

#### Сущность «Пользователь» (User)

| Атрибут (логический) | Смысл |
|----------------------|--------|
| id | Идентификатор |
| username | Логин |
| email | Электронная почта (уникальная) |
| role | Роль: участник / организатор / жюри |
| organization | Организация (текст) |
| theme | Тема интерфейса (тёмная / светлая) |
| … | При необходимости: имя, фамилия, флаги активности (из базового класса пользователя Django) |

**Связи:**

| От | К | Кратность | Пояснение |
|----|---|-----------|-----------|
| Пользователь | Соревнование | 1 : N | Роль **создатель** (`created_by`) |
| Пользователь | Участие | 1 : N | Участник турнира |
| Пользователь | Решение | 1 : N | Автор отправки |
| Пользователь | Объявление | 1 : N | Автор объявления (опционально NULL) |
| Пользователь | Событие оценивания | 1 : N | Кто изменил оценку |
| Пользователь | Уведомление | 1 : N | Получатель in-app уведомлений |
| Пользователь | Хакатон | 1 : N | Организатор-создатель |
| Пользователь | Регистрация на хакатон | 1 : N | |
| Пользователь | Участник команды хакатона | 1 : N | |

#### Сущность «Уведомление» (InAppNotification)

| Атрибут | Смысл |
|---------|--------|
| id | |
| kind | Тип: объявление / оценка / системное |
| title, body, link | Содержание и ссылка |
| read_at, created_at | Прочитано, создано |

**Связь:** Пользователь 1 — N Уведомление.

### 3.2. Соревнования по задачам (`competitions`)

#### Справочник «Предмет» (Subject)

| Атрибут | Смысл |
|---------|--------|
| id, name, slug, order | Название, код, порядок в списке |

**Связь:** Предмет 1 — N Задача (необязательная: задача может быть без предмета).

#### Справочник «Тип задания» (TaskType)

| Атрибут | Смысл |
|---------|--------|
| id, name, slug, order | Название, код (multiple_choice, match, …) |
| auto_partial_enabled, partial_weight_percent | Настройки автопроверки |

**Связь:** Тип задания 1 — N Задача.

#### Сущность «Соревнование» (Competition)

| Атрибут | Смысл |
|---------|--------|
| id, title, description | Карточка турнира |
| audience, level | Целевая аудитория, уровень |
| status | Черновик → опубликовано → регистрация → идёт → завершено |
| start_time, end_time | Окно проведения |
| min_age, max_age, max_participants | Ограничения |
| created_at, updated_at | Служебные даты |

**Связи:**

| От | К | Кратность |
|----|---|-----------|
| Пользователь | Соревнование | 1 : N (создатель) |
| Соревнование | Задача | 1 : N |
| Соревнование | Участие | 1 : N |
| Соревнование | Объявление | 1 : N |

#### Сущность «Задача» (Task)

| Атрибут | Смысл |
|---------|--------|
| id, title, description, order, max_score | Условие и балл |
| expected_output | Эталон (автопроверка) |
| allow_multiple_answers | Несколько вариантов ответа |
| opens_at, closes_at | Расписание выдачи и приёма |
| organizer_material | Материал организатора (файл) |

**Связи:**

| От | К | Кратность |
|----|---|-----------|
| Соревнование | Задача | 1 : N |
| Предмет | Задача | 1 : N (0..1 со стороны задачи) |
| Тип задания | Задача | 1 : N (0..1) |
| Задача | Вариант ответа | 1 : N |
| Задача | Пара соответствий | 1 : N |
| Задача | Решение | 1 : N |

#### Сущность «Вариант ответа» (TaskOption)

| Атрибут | Смысл |
|---------|--------|
| id, text, is_correct, order | Текст варианта, признак правильности |

**Связь:** Задача 1 — N Вариант ответа (для типа «выбор варианта»).

#### Сущность «Пара соответствий» (TaskMatchingPair)

| Атрибут | Смысл |
|---------|--------|
| id, order, left_text, right_text | Строка задания «установить соответствие» |

**Связь:** Задача 1 — N Пара соответствий (для типа `match`).

#### Сущность «Участие» (Participation) — связка M:N

| Атрибут | Смысл |
|---------|--------|
| id | |
| registered_at | Дата регистрации |

**Связи:** Пользователь N — M Соревнование **через** Участие; на диаграмме — две линии 1:N к центральному прямоугольнику **Участие**.  
**Ограничение (логическое):** пара (пользователь, соревнование) **уникальна** — отразить подписью `{Уникально: пользователь + соревнование}`.

#### Сущность «Решение» (Solution)

| Атрибут | Смысл |
|---------|--------|
| id, content | Текст / код ответа |
| attachment | Файл решения (опционально) |
| submitted_at | Время отправки |
| status | На проверке / зачтено / отклонено |
| score, comment | Баллы и комментарий жюри/организатора |

**Связи:**

| От | К | Кратность |
|----|---|-----------|
| Пользователь | Решение | 1 : N |
| Задача | Решение | 1 : N |
| Решение | Событие оценивания | 1 : N |

*На логической схеме допустимо не указывать, что у одного пользователя по одной задаче может быть **несколько** попыток (если в коде разрешено) — уточнить в тексте ПЗ.*

#### Сущность «Событие оценивания» (SolutionGradeEvent)

| Атрибут | Смысл |
|---------|--------|
| from_status, to_status, from_score, to_score | Было / стало |
| from_comment, to_comment, note | Комментарии, причина |
| created_at | Когда изменено |

**Связи:** Решение 1 — N Событие оценивания; Пользователь 1 — N (кто оценил).

#### Сущность «Объявление» (Announcement)

| Атрибут | Смысл |
|---------|--------|
| title, body, is_pinned | Текст, закрепление |
| created_at, updated_at | |

**Связи:** Соревнование 1 — N Объявление; Пользователь 1 — N Объявление (автор).

### 3.3. Хакатоны (`hackathons`) — отдельная зона на рисунке

#### Сущность «Хакатон» (Hackathon)

| Атрибут | Смысл |
|---------|--------|
| title, slug, tagline, description | Карточка мероприятия |
| audience, level, format, venue | Аудитория, формат, площадка |
| status | Черновик … завершён |
| registration_opens_at / closes_at, starts_at, ends_at | Расписание |
| min_age, max_age, max_teams | Ограничения |
| min_team_size, max_team_size | Размер команды |
| tracks, prizes, rules, results_summary | Треки, призы, правила, итоги |

**Связи:** Пользователь 1 — N Хакатон (создатель); Хакатон 1 — N Команда; 1 — N Регистрация; 1 — N Сообщение чата.

#### Сущность «Регистрация на хакатон» (HackathonRegistration)

| Атрибут | Смысл |
|---------|--------|
| team_name, looking_for_team, comment | Заявка |
| registered_at | |

**Связи:** Пользователь 1 — N; Хакатон 1 — N; Команда 0..1 — N (участник может быть без команды).  
**Уникальность:** пользователь + хакатон.

#### Сущность «Команда хакатона» (HackathonTeam)

| Атрибут | Смысл |
|---------|--------|
| name, created_at | Название, дата создания |

**Связи:** Хакатон 1 — N Команда; Пользователь 1 — N (создатель команды); Команда 1 — N Участник команды.  
**Уникальность:** имя команды в рамках одного хакатона.

#### Сущность «Участник команды» (HackathonTeamMember)

| Атрибут | Смысл |
|---------|--------|
| role | Капитан / участник / заявка |
| joined_at | |

**Связи:** Хакатон 1 — N; Команда 1 — N; Пользователь 1 — N.  
**Уникальность:** пользователь + хакатон (один пользователь — одна роль в одном хакатоне).

#### Сущность «Сообщение чата хакатона» (HackathonChatMessage)

| Атрибут | Смысл |
|---------|--------|
| team_name, channel, body, created_at | Команда, канал (команда / оргкомитет), текст |

**Связи:** Хакатон 1 — N; Пользователь 1 — N (автор).

## 5. Рекомендуемая компоновка на листе (draw.io / diagrams.net)

### Вариант A — один лист (альбом A3 или A4)

1. **Слева вверху:** **Пользователь** — центральный узел с исходящими связями.
2. **Центр:** цепочка **Соревнование → Задача → (Вариант ответа | Пара соответствий)**; снизу от соревнования — **Участие**, **Объявление**; от задачи — **Решение → Событие оценивания**.

4. **Справа:** блок **Хакатон** с командой, регистрацией, чатом (можно рамкой «Подсистема хакатонов»).
5. **Снизу у пользователя:** **Уведомление**.

## 6. Пошаговая работа (этапы 0–8)

### Этап 0. Подготовка

1. Прочитать абзац **2.3.2** в пояснительной записке и таблицы **3–4** (входные/выходные данные).

4. Согласовать с руководителем: **хакатоны** на том же рисунке или отдельно.

**Выход:** черновик списка сущностей (раздел 3 настоящей инструкции).

### Этап 1. Прямоугольники сущностей

Нарисовать **все** сущности из разд. 3 без линий. Внутри каждой — **3–8 главных атрибутов**; полный перечень — в таблице для ПЗ (см. этап 6).

### Этап 2. Связи и кратности

По таблице разд. 4 провести линии. Для **Участие** и регистраций M:N — **ассоциативная сущность** в центре.

### Этап 3. Ограничения целостности (логический уровень)

Подписи у связей или в легенде:

- уникальность пары (пользователь, соревнование);
- уникальность (пользователь, хакатон);
- уникальность (хакатон, название команды);
- каскадное удаление **не рисуют** на логической схеме (это физический уровень) — при необходимости сноска «при удалении соревнования удаляются задачи» как **бизнес-правило**.

В углу листа:

- расшифровка кратностей;

- «Пунктиром — опциональная связь (0..1)».

### Этап 6. Текст к рисунку в ПЗ (шаблон)

Вставить после рисунка 29 (адаптировать формулировки):

> Логическая схема данных (рис. 29) отражает основные сущности WEB-приложения CompetitionHub: **Пользователь** с ролями участника, организатора и жюри; **Соревнование** и связанные **Задачи** с справочниками **Предмет** и **Тип задания**; варианты ответа и пары соответствий для автоматизируемых типов заданий; ассоциативную сущность **Участие**, реализующую регистрацию на турнир; **Решение** участника и журнал **Событие оценивания**; **Объявление** организатора и **Уведомление** в личном кабинете. ### Этап 8. Чеклист перед сдачей

## 8. Типичные ошибки

1. **Смешение логической и физической схемы** на одном рисунке (типы `VARCHAR`, имена таблиц `accounts_user`).
2. **Прямая связь** пользователь—соревнование без сущности **Участие**.
3. **Отсутствие** **Событие оценивания** при наличии в коде и в сценариях жюри.
4. **Один рисунок** с 40 атрибутами в каждом блоке — нечитаемо; вынести детали в таблицу в тексте.
5. **Хакатон** нарисован как подтип **Соревнование** одной сущностью — в проекте это **разные** модели (`Competition` ≠ `Hackathon`).
6. **Вариант ответа** привязан к **Соревнованию**, а не к **Задаче**.

## 9. Связь с физической схемой и приложением Г
