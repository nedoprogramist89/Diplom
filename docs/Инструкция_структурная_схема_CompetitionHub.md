# Инструкция: структурная схема программы (CompetitionHub)

Документ для подготовки **рисунка к подразделу 2.3.4** пояснительной записки (в ПЗ — аналог «Рисунок 31 – Структурная схема»): **из каких частей состоит программа**, как они **вложены друг в друга** и **связаны по составу**, без детализации потоков данных (это DFD) и без пошагового обмена запросами (это [схема взаимодействия компонентов](Инструкция_схема_взаимодействия_компонентов_CompetitionHub.md)).

---

## 1. Цель структурной схемы и отличия от других рисунков

| Вид схемы | Вопрос, на который отвечает |
|-----------|-------------------------------|
| **Структурная** | *Из чего программа собрана?* Иерархия **модулей / пакетов / слоёв**, состав и вложенность. |
| Схема взаимодействия компонентов | *Кто кого вызывает при работе?* Маршрутизатор, представления, БД, шаблоны. |
| Функциональная (2.3.5) | *Что делает система по ролям/функциям?* Пользователь, организатор, жюри и т.д. |
| Диаграмма классов | *Какие классы, поля, связи ООП?* UML на уровне классов. |
| Логическая / физическая схема данных | *Какие сущности и таблицы?* ER и таблицы СУБД. |

Структурная схема для **Django-проекта** обычно строится как **дерево**: корень — проект целиком, ветви — приложения и общие ресурсы, **листья уровня 3** — конкретные модули `.py` и логические узлы (`migrations/`, `templatetags/`). Ниже уровень 3 расписан **подробно** по репозиторию CompetitionHub — можно переносить подписи на рисунок выборочно или вынести часть на **второй лист** («детализация приложения competitions»).

## 2. Требования методических материалов (кратко)

В методичке для **структурной схемы программы** указано:

- отразить **принцип построения**: модули, процедуры/функции, **назначение и взаимосвязь**;
- представление — **схема и/или таблица**;
- при **ООП** дополнительно: иерархия на **программном уровне**, разграничение объектов на **используемые**, **доработанные (перекрытые)** и **собственные (автора)**;
- поясняющая таблица для модулей автора: **номер**, **название модуля**, **описание модуля**;
- рекомендуется **не менее трёх уровней** детализации (укрупнённый → средний → детальный при необходимости).

Имеет смысл **один рисунок** (укрупнённая иерархия) + **таблица модулей** в тексте ПЗ (как таблица 6 «Характеристики модулей» или отдельная «Пояснение к рис. 31»).

## 3. Рекомендуемые три уровня для CompetitionHub

### Уровень 1 (корень дерева)

Один блок, например: **«WEB-приложение CompetitionHub»** или **«Проект Django CompetitionHub»**.

### Уровень 2 (крупные части)

Разбить на осмысленные подсистемы (прямоугольники **внутри** корня):

| Блок уровня 2 | Содержание |
|---------------|------------|
| **Конфигурация проекта** | Пакет `config` — настройки, корневые URL, админ-сайт, API-агрегатор, WSGI/ASGI. |
| **Приложение accounts** | Учётные записи, профиль, уведомления, API пользователя. |
| **Приложение competitions** | Соревнования, задачи, решения, результаты, API турнира. |
| **Приложение hackathons** | Хакатоны, команды, чат (если включено в объём работы). |
| **Шаблоны и статика** | Каталоги `templates/`, `static/` (общие ресурсы интерфейса). |
| **Служебные файлы проекта** | `manage.py`, `requirements.txt`, при необходимости `.env.example` — по указанию руководителя. |

При **курсовом** объёме блок **hackathons** можно объединить с **competitions** подписью «прикладные приложения» или оставить отдельно, если в задании хакатоны явно выделены.

### Уровень 3 — подробный состав по пакетам репозитория

Ниже перечислены модули, которые **имеет смысл показать** на структурной схеме как дочерние узлы соответствующего блока уровня 2. Миграции обычно **сворачивают** в один прямоугольник **`migrations/`** (без перечисления каждого файла), если иначе рисунок перегружен.

#### 3.1. Пакет `config` (конфигурация проекта)

| Модуль | Назначение (подпись на схеме) |
|--------|-------------------------------|
| `settings.py` | Регистрация `INSTALLED_APPS`, `DATABASES`, `STATIC`/`MEDIA`, шаблоны, JWT, CORS, локализация. |
| `urls.py` | Корневой `urlpatterns`: `admin/`, `accounts/`, `api/`, `hackathons/`, корень → соревнования. |
| `api_urls.py` | Подключение `accounts` и `competitions` API под префиксом `/api/`. |
| `admin_site.py` | Кастомный `AdminSite` и регистрация моделей панели управления. |
| `wsgi.py` | Точка входа WSGI для боевого/тестового сервера. |
| `asgi.py` | Точка входа ASGI (асинхронный сервер при необходимости). |
| `__init__.py` | Пакет конфигурации (при укрупнении можно не выделять отдельным блоком). |

#### 3.2. Приложение `accounts`

| Модуль | Назначение |
|--------|------------|
| `models.py` | Пользователь `User`, внутрисистемные уведомления `InAppNotification`. |
| `views.py` | Вход, регистрация, профиль, смена пароля, сброс по email, уведомления, публичный профиль. |
| `urls.py` | Маршруты учётной записи. |
| `forms.py` | Формы регистрации и редактирования профиля. |
| `admin.py` | Отображение пользователя в админке. |
| `apps.py` | Конфигурация приложения `AppConfig`. |
| `serializers.py` | Сериализация пользователя для REST. |
| `api_views.py` | JWT, регистрация, текущий пользователь. |
| `api_urls.py` | Маршруты API учётных записей. |
| `context_processors.py` | Тема оформления, уведомления в шаблонах. |
| `notifications.py` | Вспомогательные функции рассылки уведомлений. |
| `migrations/` | Изменения схемы БД для `accounts` (одним блоком). |

#### 3.3. Приложение `competitions`

| Модуль | Назначение |
|--------|------------|
| `models.py` | Предметы, типы задач, соревнование, задача, варианты, соответствия, участие, решение, объявления и др. |
| `views.py` | Главная, список/карточка соревнования, задачи, регистрация, отправка и оценка решений, результаты, CSV, архив, сертификат. |
| `urls.py` | Веб-маршруты соревнований. |
| `forms.py` | Формы соревнований, задач (варианты, соответствия), отправка решения, вложения. |
| `admin.py` | Регистрация моделей соревнований в панели. |
| `permissions.py` | Проверки прав по ролям и владельцу турнира, миксины для CBV. |
| `serializers.py` | Сериализаторы API с ограничением полей по ролям. |
| `api_views.py` | REST: соревнования, задачи, участие, решения, оценки. |
| `api_urls.py` | Маршруты API соревнований. |
| `signals.py` | Уведомления при смене статуса соревнования. |
| `context_processors.py` | Права на создание соревнований в шаблонах. |
| `auto_grade.py` | Автоматическая проверка ответов, частичный балл. |
| `matching_utils.py` | Логика заданий типа «соответствие». |
| `task_visibility.py` | Видимость задачи и приём решений по расписанию. |
| `schedule.py` | Этапы (фазы) соревнования для карточки турнира. |
| `apps.py` | Конфигурация приложения. |
| `templatetags/competitions_extras.py` | Пользовательские теги и фильтры шаблонов. |
| `management/commands/runserver.py` | Кастомная команда `runserver` (порт из окружения). |
| `migrations/` | Все файлы миграций схемы `competitions` — **одним** дочерним узлом. |

#### 3.4. Приложение `hackathons`

| Модуль | Назначение |
|--------|------------|
| `models.py` | Хакатон, регистрация, команда, члены, сообщения чата и т.д. |
| `views.py` | Список, карточка, CRUD, этапы, команды, чат, заявки, капитанство. |
| `urls.py` | Маршруты раздела хакатонов. |
| `forms.py` | Формы хакатона, регистрации, чата, команды. |
| `admin.py` | Регистрация моделей в админке. |
| `permissions.py` | Права на создание и редактирование хакатона. |
| `signals.py` | Уведомления при смене статуса хакатона. |
| `schedule.py` | Фазы и подсказки по календарю для шаблонов. |
| `context_processors.py` | Право создавать хакатоны. |

| `migrations/` | Миграции `hackathons` — одним блоком. |

#### 3.5. Каталоги `templates/` и `static/`

На схеме уровня 3 для интерфейса достаточно **двух узлов** с кратким пояснением в подписи или раскрытия на четвёртом уровне (по согласованию с руководителем):

| Узел | Содержание (для подписи или вложенных блоков) |
|------|-----------------------------------------------|
| `templates/` | `base.html`; подкаталоги `accounts/`, `competitions/`, `hackathons/` — страницы по разделам. |
| `static/` | `css/` (в т.ч. темы), `js/` — статика интерфейса. |

#### 3.6. Служебные файлы в корне репозитория

| Файл | Назначение |
|------|------------|
| `manage.py` | CLI Django: `runserver`, `migrate`, `createsuperuser` и др. |
| `requirements.txt` | Зависимости Python для развёртывания. |

**Как наносить на рисунок.** Вариант А: под каждым приложением перечислить **все** строки таблиц 3.1–3.4 мелким шрифтом или колонками. Вариант Б: на основном листе — только «тяжёлые» узлы (`views`, `models`, `forms`, `permissions`, `api_*`), остальное вынести на **вставку** или второй рисунок «Состав приложения competitions». Вариант В: сгруппировать узлы **«веб-слой»** (`urls` + `views` + `forms`), **«данные»** (`models` + `migrations`), **«API»** (`serializers` + `api_views` + `api_urls`), **«инфраструктура приложения»** (`signals`, `context_processors`, `schedule`, `templatetags`, `management`).

**Не дублируйте** на структурной схеме содержимое **диаграммы классов**: здесь не нужны поля и методы классов — только **файлы / пакеты** и иерархия.

## 4. Классификация модулей для подписи «используемые / доработанные / свои»

Чтобы выполнить формулировку методички про ООП, на схеме или в легенде к рисунку отметьте цветом или префиксом:

| Категория | Примеры в CompetitionHub |
|-----------|---------------------------|
| **Используемые** | Фреймворк Django, DRF, библиотеки (на схеме часто **не раскрывают** до файлов, а одним блоком «Django / DRF» вне вашего кода). |
| **Доработанные (перекрытые)** | Кастомный пользователь `User(AbstractUser)`, кастомный `AdminSite` в `config/admin_site.py`, расширение стандартных классов представлений миксинами. |
| **Собственные (автора)** | Основная логика в `competitions/views.py`, `competitions/models.py`, `competitions/forms.py`, `competitions/permissions.py`, `accounts/views.py` и т.д. |

На **укрупнённой** схеме достаточно легенды из трёх условных обозначений; в **таблице** к рисунку — явная колонка «Категория» для строк с модулями автора.

## 6. Пошаговый порядок построения рисунка

1. Зафиксировать **корень** (уровень 1) — название продукта из темы ПЗ.
2. Нарисовать **уровень 2**: `config`, `accounts`, `competitions`, (+ `hackathons`), `templates`/`static`, при необходимости «точка входа» `manage.py`.
3. Для **каждого** блока уровня 2 развернуть **уровень 3** по таблицам **п. 3.1–3.6** (полный список или варианты группировки **3.6** — А / Б / В); при нехватке места — второй лист «Детализация: competitions».
4. Связи на структурной схеме показывают **состав и иерархию** («входит в состав», «относится к»), а не обязательно вызовы во времени. Допустимы пунктирные границы «логический слой: представления» внутри приложения.
5. Оформить **таблицу** (п. 5) и вставить в ПЗ после рисунка 31 с подписью «Пояснение к структурной схеме» при необходимости.
6. Сверить с **`config/settings.py`** → `INSTALLED_APPS`, чтобы на схеме не было «лишнего» приложения и не пропущено то, что реально подключено.

## 8. Проверка перед сдачей

- [ ] Есть **не менее трёх** уровней вложенности; уровень 3 покрывает **основные модули** из п. 3.1–3.6 (или их **укрупненные группы** по варианту 3.6 — при этом в таблице к рисунку перечислены конкретные файлы).
- [ ] Структурная схема **не дублирует** функциональную (роли) и **не заменяет** схему взаимодействия компонентов.
- [ ] Имена пакетов и файлов совпадают с **репозиторием** и с **таблицей модулей** в ПЗ.
- [ ] В тексте подраздела **2.3.4** есть отсылка к **рисунку 31** и краткое словесное пояснение дерева (3–5 предложений).
